# Data Model: Database Initialization & Auth Handshake

## Entity: User

Represents a registered account holder.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID/string | Yes | Primary key. Generated by the database/Prisma layer. |
| `email` | string | Yes | Unique, normalized lowercase email. Used for login. Do not write to audit metadata. |
| `passwordHash` | string | Yes | Hash of the user's password. Plaintext password is never persisted. |
| `status` | enum | Yes | `ACTIVE` or `INACTIVE`; default `ACTIVE`. |
| `createdAt` | datetime | Yes | Account creation timestamp. |
| `updatedAt` | datetime | Yes | Last account record update timestamp. |
| `lastLoginAt` | datetime nullable | No | Set after successful login. |

### Validation Rules

- Email must be syntactically valid and capped at an implementation-defined maximum length.
- Email uniqueness is enforced by a database unique constraint.
- Password must be at least 8 characters and include uppercase, lowercase, digit, and special character.
- Password is accepted only at registration/login boundaries and never stored outside the password hashing operation.

### Relationships

- One `User` can have many `AuditLog` records.

### State Transitions

```text
ACTIVE -> INACTIVE
INACTIVE -> ACTIVE
```

The first feature only requires `ACTIVE` accounts. `INACTIVE` is reserved for future administrative or compliance flows.

## Entity: AuditLog

Represents a security/audit event for authentication.

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUID/string | Yes | Primary key. |
| `userId` | UUID/string nullable | No | Present when a user is known. Null for unknown failed login attempts. |
| `action` | enum | Yes | `AUTH_REGISTERED`, `AUTH_LOGIN_SUCCEEDED`, `AUTH_LOGIN_FAILED`, `AUTH_LOGOUT`, `AUTH_LOCKED_OUT`, `AUTH_TOKEN_REJECTED`. |
| `resourceType` | string | Yes | Use `auth` for this feature. |
| `resourceId` | string nullable | No | User ID or auth event ID when applicable. |
| `metadata` | JSON | Yes | Safe metadata only: result, reason code, lockout duration, IP hash, user agent hash, trace ID. No email, password, raw token, or card data. |
| `traceId` | string | Yes | Request trace identifier. |
| `correlationId` | string | Yes | Cross-service correlation identifier. |
| `createdAt` | datetime | Yes | Event timestamp. |

### Validation Rules

- `metadata` must not contain email, plaintext credentials, raw JWTs, or raw IP addresses.
- Every registration, login success, login failure, logout, and lockout event creates one audit record.

### Relationships

- Many `AuditLog` records may reference one `User`.
- Failed attempts for unknown accounts may have no user relationship.

## Runtime State: Auth Rate Limit

Stored in Redis, not PostgreSQL.

| Key | Value | TTL | Notes |
|-----|-------|-----|-------|
| `auth:failed:{ipHash}` | integer | 15 minutes | Failed login attempts in the rolling window. |
| `auth:lockout:{ipHash}` | lockout metadata | active lockout duration | Blocks login attempts during lockout. |
| `auth:lockout-level:{ipHash}` | integer | implementation-defined | Calculates 1, 2, 4, then 8 minute lockouts. |

### State Transitions

```text
no failures
  -> failed count 1..5
  -> locked out for 1 minute
  -> locked out for 2 minutes
  -> locked out for 4 minutes
  -> locked out for 8 minutes
  -> reset after successful login
```

## Session Token

JWT issued by the backend and consumed by the frontend session layer.

| Claim | Required | Notes |
|-------|----------|-------|
| `sub` | Yes | User ID. |
| `email` | Optional | Avoid logging; include only if needed by NextAuth session display. |
| `iat` | Yes | Issued-at timestamp. |
| `exp` | Yes | 24 hours after issuance. |

### Validation Rules

- Missing, expired, malformed, or tampered tokens are rejected.
- Protected endpoints must use `JwtAuthGuard` or an equivalent guard.
- Logout clears the frontend session; no server-side token revocation is required for v1.
